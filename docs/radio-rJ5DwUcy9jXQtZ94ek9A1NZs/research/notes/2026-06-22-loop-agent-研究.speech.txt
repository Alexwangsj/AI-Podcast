今天我们来研究一个最近越来越常被提到的词：Loop Agent。

先把概念说清楚。Loop Agent 不是某一家公司的单一产品，而是一类架构模式。它的核心，是让 AI 不再只是回答一次问题，而是在一个循环里持续工作：观察当前状态，推理下一步，调用工具，读取结果，再决定是否继续。直到任务完成，或者触发停止条件。

如果用一句话概括，就是：Loop Agent 等于大模型，加工具，加状态，加验证器，再加一个停止条件。

为什么这件事重要？因为它解释了 chatbot 和 agent 的根本差别。普通聊天机器人是一次输入，一次输出。你问它一个问题，它回答，然后结束。Agent 则要面对多步骤任务。比如“读一下这个代码库，找出测试失败原因，修复它，跑测试，再提交”。这个任务不可能靠一次回答完成。它必须能读文件、运行命令、观察错误、修改方案，再继续下一轮。

Oracle 有一篇讲 agent loop 的文章，把这个循环拆成五步：感知、推理、计划、行动、观察。感知是读用户输入、工具结果、错误信息或环境状态。推理是判断现在该做什么。计划是把复杂目标拆成子任务。行动是调用工具、查数据库、执行代码、访问 API。观察是看行动是否成功，然后把结果带回下一轮。

听起来很复杂，但最小伪代码其实很简单：

只要任务没完成，就调用模型。如果模型要用工具，就执行工具，把工具结果塞回上下文。如果模型不再需要工具，就返回最终结果。

这就是 agent loop 的内核。

OpenAI Agents SDK 文档里也有类似描述。一次 SDK run 是一个应用层 turn。runner 会循环：先调用当前 agent 的模型，然后检查输出。如果模型产生工具调用，就执行工具并继续。如果模型把任务 handoff 给另一个 specialist，就切换 agent 继续。如果模型输出最终答案并且没有后续工具工作，才返回结果。也就是说，tools、handoffs、approvals、streaming 都是在这个 loop 上建起来的。

不过，Loop Agent 不只有一种形态。我们可以分三层看。

第一层，是内层工具循环。

这是最常见的 loop。模型决定调用搜索工具、文件工具、浏览器工具、数据库工具或代码执行工具。工具返回结果后，模型再根据观察结果继续推理。AI SDK 的 ToolLoopAgent 就是这个方向。它和单步 `generateText` 不同，能够多步调用工具，直到达到停止条件或者需要用户批准。

第二层，是工作流循环。

Google ADK 的 LoopAgent 是代表。它不是让模型自由决定流程，而是确定性地反复执行一组 sub-agents。比如 Writer Agent 写一版文档，Critic Agent 评论，Writer 再修改，Critic 再评论。LoopAgent 最多跑五轮，或者直到 Critic 发出“质量已经足够”的停止信号。

这个形态很适合“迭代改进”任务。比如写文档、生成 SQL、优化 prompt、修复代码、生成图片后计数检查。它比完全自主 agent 更可控，因为循环结构由程序定义，模型只负责每一步的具体内容。

第三层，是外层验证循环。

Vercel Labs 的 Ralph Loop Agent 很典型。它把 AI SDK 的工具循环放在内层，然后在外层再加一个 `verifyCompletion`。内层 agent 可能会说“我完成了”。但外层验证器会问：真的完成了吗？测试过了吗？结果符合目标吗？如果不符合，就把原因反馈给 agent，让它再跑一轮。

这个设计很接近我们实际使用 coding agent 的体验。很多时候模型自信地说修好了，但测试没过，页面没打开，或者只改了表面问题。外层 loop 的价值，就是把“模型觉得完成”变成“验证器确认完成”。

这也是最近所谓 loop engineering 的核心。Business Insider 最近报道说，一些 AI coding 领域的人开始从手写 prompt 转向设计 loop。Claude Code 的 Boris Cherny、OpenAI 工程师 Peter Steinberger 等人都在强调，不只是写一句 prompt，而是让 agent 在目标、工具、反馈和验证之间持续迭代。

但是，这里要小心一个误区：Loop Agent 不是越自主越好，也不是无限循环就聪明。

Anthropic 在 “Building effective agents” 里讲得很务实：先用最简单的架构解决问题。只有当简单 prompt、固定 workflow 或单次工具调用不够时，再引入 agent。原因很简单：agent 自主性越强，成本越高，错误也越可能累积。你需要 sandbox、guardrails、透明的 planning，以及良好的工具设计。

这里有一个很重要的词，叫 ACI，agent-computer interface。它类似 HCI，也就是人机交互。但对象从人变成了 agent。你给 agent 的工具好不好用，参数清不清楚，错误信息能不能理解，有没有示例，边界是否明确，都会直接影响它能不能完成任务。

Anthropic 举过一个很好的例子：工具如果要求模型手写复杂 diff，模型可能因为行号和格式失败；如果工具改成更接近自然代码编辑的形式，成功率会提高。这个经验对 Loop Agent 很关键。因为 loop 会把工具使用放大，工具设计里的小摩擦，会在多轮循环里变成大失败。

接下来讲 Loop Agent 的适用场景。

第一类，是步骤数无法预先确定的任务。比如调试代码、做深度研究、迁移项目、分析日志、追查生产问题。你不知道要查几次、改几处、跑几轮测试。

第二类，是中间结果会改变后续策略的任务。比如搜索到的资料会改变研究方向，测试失败会改变修复方案，日志分析会指向新的服务。

第三类，是有可验证完成条件的任务。比如测试通过、页面能打开、音频时长达标、RSS 包含新条目、SQL 查询返回正确结果。没有验证条件，loop 就容易变成模型自我感觉良好。

第四类，是错误可以通过反馈修复的任务。Loop 的价值就是失败后能调整策略。如果失败不能提供有用反馈，循环也救不了。

反过来，有些任务不适合 Loop Agent。

固定流程不适合。比如每天固定从一个 API 取数据、清洗、写入表格，这种用普通 pipeline 更稳。

低延迟场景不适合。Loop 每一轮都要模型调用、工具调用和观察，天然更慢。

高风险写操作不适合直接全自动。涉及付款、删除数据、修改权限、合规审批，必须有人审或者强权限边界。

没有预算约束也不适合。Loop Agent 很容易烧 token，尤其是多 agent、多工具、长上下文任务。必须限制最大迭代、最大成本、最大工具调用次数，以及什么时候升级给人。

再看一个产品化例子：Braintrust 的 Loop。

Braintrust Loop 是嵌入 Braintrust 产品里的 AI agent。它可以用自然语言分析 logs 和 traces，生成 SQL filters，找相似 traces，创建 datasets 和 scorers，优化 prompts，搜索文档，甚至提交 support tickets。这个例子说明，Loop Agent 很自然地进入 observability 和 evaluation 平台。因为 agent 自己也需要被观察、被评估、被调试。

这和我们做 AI 播客项目也有关系。我们的日报生成流程，其实已经有 Loop Agent 的雏形。

第一步，研究资料。第二步，写 notes 和 speech。第三步，生成 MP3。第四步，验证音频时长、RSS、metadata、归档，以及日报里不能留播报稿。第五步，如果音频太短，就把原因反馈给写稿环节，扩稿，然后重新生成。

这就是一个很实际的 loop：目标明确，工具明确，检查项明确，失败后有反馈，成功后提交发布。

如果要把它进一步产品化，我会加四个东西。

第一，显式状态文件。比如 `run.json` 记录本次任务的标题、目标时长、来源、生成文件、验证结果和失败原因。

第二，验证器独立化。不要让写稿 agent 自己判断自己是否成功，而是用脚本检查音频时长、RSS、归档、文件泄漏、链接可访问性。

第三，预算边界。比如最多重生成两次，最多抓取二十个来源，最多跑三十分钟。超过就暂停并报告。

第四，人类介入点。比如来源冲突、重大事实不确定、音频连续两次不达标、Git 推送失败，就停下来让人决定。

如果从工程实现角度看，一个生产级 Loop Agent 最好再拆成五个文件或五个模块。

第一个模块是任务规格。它不是一句“帮我做好这个”，而是结构化的目标：输入是什么，输出是什么，完成标准是什么，不能做什么，预算是多少。目标越模糊，loop 越容易空转。

第二个模块是工具层。这里要把工具做窄，而不是做大。比如不要给 agent 一个万能 shell 就完事，而是优先给它搜索、读文件、写文件、运行测试、生成报告这些边界清晰的工具。工具返回值也要适合模型阅读：错误要清楚，成功要简短，路径要绝对，状态要可机器解析。

第三个模块是记忆层。很多人以为记忆就是把所有上下文塞回 prompt。其实更稳的做法，是把状态写到文件、数据库或 trace 里。下一轮只读取关键摘要、待办事项、失败原因和验证结果。这样可以避免上下文膨胀，也能让一次失败后的恢复更可靠。

第四个模块是验证层。验证最好尽量确定性。代码任务就跑测试和类型检查；网页任务就截图和检查 DOM；播客任务就检查音频时长、RSS 和文件泄漏；数据任务就检查 schema、行数和抽样结果。只有在无法写规则时，再用另一个模型做 evaluator。

第五个模块是审计层。每一轮做了什么、用了哪些工具、花了多少 token、失败在哪里、为什么继续或停止，都要留下记录。没有审计，Loop Agent 很难被信任；出了问题也只能猜。

所以，设计 Loop Agent 的关键不是把提示词写得神乎其神，而是把循环里的每个接口变得朴素、可测、可恢复。一个好的 loop，看起来甚至有点无聊：目标清楚，工具简单，状态明确，验证严格，预算有限。正是这种无聊，让它能在真实工作里可靠运行。

最后总结一下。

Loop Agent 的本质不是一个炫酷概念，而是一个工程控制结构。它把大模型放进循环里，让模型能够根据环境反馈持续行动。但这个循环必须有边界：状态、工具、验证、预算、权限、停止条件。

内层工具循环让模型能行动。工作流循环让多个 agent 能迭代改进。外层验证循环让“模型说完成”变成“系统确认完成”。这三层合起来，才是可靠的 Loop Agent。

所以，真正值得学的不是“让 AI 无限跑”，而是设计一个能持续推进、能被验证、能安全停下来的循环。
